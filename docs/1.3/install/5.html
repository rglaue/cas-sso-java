Frequently Asked Questions

General

What Blackboard versions is this module compatible with?

  • Blackbaord 8.0.x, tested
  • [Blackboard] WebCT 4.3, untested but should work.
  • The module does not work with Blackboard 9.

Configuration

I see more than one source.id and source.name. How do I make sure I am using the correct ones.

You are probably seeing one set for the Domain and another set for the Institution. You should only use the set for the Institution. This corresponds to the fact that you should be configuring the module on the Institution admin level.
Follow the instructions in the section Find the Source id and name for your institution, and perform both methods to find the source.id and source.name sets. The information found by executing `siapi.sh` and through the Web Admin interface should be identical.

There are two administrative levels, and three configuration levels, where am I configuring the module?

Blackboard 8 has two system administrative levels: Server and Domain. But has three configuration levels: Server, Domain and Institution. (There is also an Instituion Admin level, but that level is accessible from the Domain Admin super account. When it is stated "system administrative levels", it is meant access by the default Blackboard super accounts. There are two Blackboard super accounts: Server Admin and Domain Admin.)

The module must be activated on all three configuration levels, and configured specifically on the Institution configuration level. To activate, you must log in to the Server Admin level to activate the module on the Server Configuration level. Then you must logout of the Server Admin level, and login to the Domain Admin level to activate the module on the Domain Configuration level as well as the Institution Configuration level.

If you are configuring a Default SSO Institution in Blackboard, you must login to the Domain Admin level, and configure it on the Domain Configuration level. This is because when a user accesses a Blackboard 8 configuration they are accessing it via a configured Domain. So that is where the Default SSO Institution is configured. When you access a configured Domain without indication of what Institution, the Domain defaults to the Default SSO Institution.

How to access the Institution settings configuration within a Domain Admin login session:

  1. You login to the Domain Admin realm
  2. However DO NOT click on the "settings" (that is the settings for the Domain Configuration).
  3. FIRST click on your Institution to enter the Institution's specific configuration screen.
  4. Then click on the "settings" link within this Institution configuration screen.

Technical Implementation

I am receiving a "BAD_CERTIFICATE" error in the logs when the module attempts to validate a CAS ticket.

The CAS SSO Module is unable to connect to a CAS Server via SSL to validate tickets if the CAS Server is using a wildcard SSL Certificate. This is a SSL Certificate with a common name like '*.domain.com'.
When the module attempts a connection like this, the following error occurs: 

 javax.net.ssl.SSLKeyException: FATAL Alert:BAD_CERTIFICATE - A corrupt
 or unuseable certificate was received.
        at edu.wiu.sso.weblogicHttpsUrlRequest.urlRequest(weblogicHttpsUrlRequest.java:145)
The reason is due to an issue with Web Logic that has not been resolvable. The the CAS Client library, used by the module, implements a NULL Name Validator for the Web Logic SSL Configuration. However the SSL Context still does not recognize the common name if it is a wild card.

The workaround, assuming your CAS Server is running on Tomcat, is to configure another SSL Listen Socket to listen on another port (like 8443) with a self-signed SSL Certificate. Then configure Tomcat to only allow the Blackboard servers to access this SSL Listen Socket. And of course configure the CAS SSO Module's CAS Server Validation URL to connect to the CAS Server at this port.

I am sure I have everything configured correctly, but the module is not being activated.

Perhaps you can login as the server admin in your browser, but cannot activate the module via the CAS login procedure. If the module is configured correctly, the issue is likely due to an issue with browser cookies. You should clear out your browser cookies and reattempt the CAS login procedure without logging in to Blackboard with any other method.

We just upgraded Blackboard 8 from 8.0.x to `> 8.0.x` and I can no longer login.

Clear your browser cookies. This is a known issue with Blackboard 8. Something changes in the Blackboard update that prevents you from being able to login to Blackboard if you have cookies in your browser from a previous session before Blackboard was updated.
